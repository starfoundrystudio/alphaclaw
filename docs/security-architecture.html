Target security architecture

AlphaClaw isolated runtime with brokered credentials

The target state keeps the agent runtime on a dedicated CPX31 host, keeps raw credentials on a separate CPX11 broker, constrains ingress through Tailscale, constrains egress through Agent Vault, and adds host-level egress inspection for any bypass.

Visible Firewall Boundary

CPX31 and CPX11 each have a dedicated Hetzner Firewall boundary, with ingress and planned egress policy called out directly.

AlphaClaw management traffic targets Agent Vault API port 14321; OpenClaw runtime traffic targets proxy port 14322.

No Wrapper Layer

AlphaClaw provisions OpenClaw proxy config and CA trust directly instead of launching the gateway through another wrapper.

Direct Egress Denied

The intended posture is brokered outbound traffic only, with PipeLock and host firewall policy catching bypass attempts.

Control layers

What Each Layer Prevents

The security model is strongest when every control has a different failure mode. Identity, firewall policy, credential brokering, and egress inspection should each catch something the others miss.

Ingress Minimization

Reduces the remote attack surface on the runtime host.

  • IDedicated CPX31 per instance.
  • ITailscale Serve for control UIs.
  • IFunnel only for allowlisted callbacks.

Admin Access

Moves SSH identity and authorization into the tailnet policy.

  • ITailscale SSH required.
  • IPublic SSH blocked by firewall.
  • ATailnet SSH policy must stay narrow.

Credential Isolation

Keeps raw secrets away from the agent runtime.

  • PAgent Vault on separate CPX11.
  • PRuntime receives proxy-role token only.
  • PCredentials injected by service policy.

Exfiltration Resistance

Limits and monitors outbound traffic from the runtime host.

  • PCPX31 egress allowlist to CPX11.
  • PPipeLock scans outbound payloads.
  • XDirect upstream internet denied.

Security review

Design Risks To Close

These are not reasons to avoid the architecture; they are the places where a strong diagram can hide operational edge cases.

  • 1 Outbound firewall rules need both cloud and host layers. Hetzner Cloud Firewalls are allow-by-default for outbound traffic unless outbound rules are defined. Add host firewall enforcement too, because local processes, overlays, and future network changes can surprise a cloud-only model.
  • 2 Do not block Tailscale's own control/data traffic. CPX31 still needs Tailscale connectivity for Serve, Funnel, and SSH. Egress policy should explicitly preserve required Tailscale traffic while blocking ordinary direct internet egress.
  • 3 Keep Agent Vault management and proxy tokens separate. OpenClaw should get only a revocable proxy-scoped token. AlphaClaw may need a management token for credential UI operations, but that token should never be injected into the agent runtime.
  • 4 CPX31 will still hold operational secrets. The goal is to remove upstream API credentials from the agent runtime, not to make CPX31 secret-free. Setup passwords, Tailscale state, proxy-role tokens, and any AlphaClaw management token still need strict filesystem permissions and rotation paths.
  • 5 Agent Vault proxy exposure is the critical boundary. The proxy port should be private to CPX31. If it is exposed publicly, token theft, brute force attempts, or unintended use of the broker become much more serious.
  • 6 Funnel path allowlists must be exact and tested. Broad prefixes can accidentally expose setup or gateway routes. Treat every new webhook/OAuth path as a change that needs review, tests, and a negative-path check.
  • 7 Proxy policy only covers traffic that reaches the proxy. Raw sockets, UDP, package managers, and direct SDK traffic must be controlled by firewall policy, proxy-aware runtime config, or both. Include IPv4 and IPv6 rules in the deny-by-default design.
  • 8 PipeLock is detection, not primary containment. It is valuable as a catch layer, but the primary protection should remain no raw secrets on CPX31 plus enforced egress through Agent Vault.
